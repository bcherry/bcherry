Thoughtful Web Applications

by Ben Cherry - @bcherry

TechTalksTO - March 29th, 2011

who am I?

I lead a front-end engineering team at Twitter

I worked on #newtwitter

why am I giving this talk?

I'm passionate about quality in front-end engineering

thought·ful

adj. showing consideration for the needs of other people

thoughtfulness is a prerequisite

to developing great applications

think about your users

every single one of them

whether you have 200 or 200 million

you won't always be able to get it right

but that doesn't mean you don't have to think about it

what is a web application?

we now have desktop-class applications on the web

ex: gmail, #newtwitter, facebook, aviary

how is this different?

ajax means we no longer share the original browser/server contract

users don't realize this

it's our responsibility to behave

what do we need to think about?

performance, testing, accessibility, browser support,

url structure, mobile devices, and more!

browser support

the first thing to think about

aim for a good experience

not necessarily consistent

example: rounded corners

use CSS border-radius

no support in IE6/7/8, but this is ok

don't be lazy, include all vendor prefixes you can

practicality

you can't test and tweak for every browser

unless you have an absurd number of engineers

establish browser grades

  • ex. A-Grade: Latest IE, Chrome, Firefox, Safari, Opera

  • ex. B-Grade: Popular legacy versions (IE7/8, Firefox 3)

  • ex. C-Grade: Internet Explorer 6

establish support for each grade

  • ex. A-Grade: All features, near consistent experience

  • ex. B-Grade: All features, slightly degraded experience

  • ex. C-Grade: Critical features, degraded experience

C-grade can lose features, but it can not be broken

good markup matters

focus on clean, clear, consistent markup

invest in CSS

constantly refine, clarify, and consolidate stylesheets

provide tools

developers should have all browsers easily accessible

establish processes

make browser testing baked-in to your review process

Non-JavaScript Users

this can get tricky

should you support no-JS?

maybe not!

but you should not be broken

reasons to support no-JS

  • so your content can be indexed

  • so your content is available to all

probably 1-2% of your visitors have JS off

a good approach

  • assume JS in initial page

  • but serve meta-redirect via <noscript> in <head>

  • send to foo.com?_noscript=1

  • on server, set noscript cookie

  • from now on, render all content on server

or redirect to your mobile site

this can can be a good compromise

accessibility

you must design for all

listen to a screen reader

you should understand how these view your app

clear markup

simple, semantic markup goes a long way

The URL

the most unique part of web applications

good principles for URLs

  • easy to guess

  • easy to speak

  • easy to understand

twitter.com/bcherry

  • ✓ easy to guess

  • ✓ easy to speak

  • ✓ easy to understand

this is a good URL

twitter.com/user?screen_name=bcherry

  • ✗ easy to guess

  • ✗ easy to speak

  • ✓ easy to understand

this is a bad URL

twitter.com/user/123456

  • ✓ easy to guess

  • ✓ easy to speak

  • ✗ easy to understand

this is a bad URL

twitt.er/bcherry

  • ✓ easy to guess

  • ✗ easy to speak

  • ✓ easy to understand

URLs like this occur too often

what about the hashbang?

much has been said on this topic lately

some background

the URL hash is for client anchors

it doesn't cause navigation when it changes

it allows bookmarking for JavaScript applications

the hashbang allows google indexing

should you use it?

it provides a better experience

1-to-1 mapping (ex. twitter.com/#!/bcherry)

encourage sharing of non-hashbang URLs

support the HTML5 History API

testing

you owe it to your users to do it

pick the right tools

I use QUnit

At Twitter, we also use Selenium

simulate as many users you can

aim for all browsers

test without JavaScript

test all functionality

if it's important to your users, test it

if it's not important, why do you have it?

integrate into your process

you should not be able to ship without passing tests

errors

a normal part of a well-designed system

don't report garbage

users don't understand or care about error codes

try not to show anything

the most graceful thing is to retry or otherwise recover

good messages

try to be friendly, apologetic, and informative

"Error 140: Status text exceeds character maximum."

this would be a bad message

"Oops! Looks like your Tweet was too long. Try to keep it under 140 characters."

this is better

messages are important

errors represent your system at its worst

good design and thoughtfulness are critical

but remember: good messages are

not an excuse for a broken system

mobile

many users will visit on a mobile device

try to provide a mobile-optimized version

this should be separate from your main app

detect platforms and advertise your native app

it's ok to automatically send to the mobile site

but make it simple to access the full site

try to include mobile devices in automated testing

performance

solve real problems

don't prematurely optimize good code away

collect performance data

discover problems before users report them

when performance degrades

tell the user

but keep trying

gmail does this right

"sending your message..."

"still working..."
